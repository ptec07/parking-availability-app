# 주차가능성예측앱 기획 초안

**작성일:** 2026-04-28  
**상태:** 기획 시작 / MVP 정의  
**목표:** 사용자가 목적지를 입력하면 주변 주차장의 “지금 주차 가능성”과 “도착 시점의 주차 성공 가능성”을 직관적으로 보여주는 앱을 만든다.

---

## 1. 문제 정의

도심 이동에서 주차는 도착 직전까지 불확실하다. 기존 지도 앱은 주차장 위치와 요금은 보여주지만, 사용자가 실제로 알고 싶은 질문은 다르다.

> “내가 20분 뒤 도착하면 여기에 댈 수 있을까?”

이 앱은 단순 주차장 검색이 아니라 **주차 가능성 예측**을 핵심 가치로 삼는다.

---

## 2. 핵심 사용자

### 2.1 1차 타깃

- 자차로 도심·상가·병원·역세권에 가는 일반 운전자
- 아이 동반 가족, 병원 방문자, 약속 시간에 늦으면 안 되는 사람
- 공영주차장을 선호하는 운전자

### 2.2 2차 타깃

- 소상공인: 매장 주변 주차난 안내
- 지자체: 공영주차장 혼잡 안내 서비스
- 배달/방문 업무자: 짧은 체류 주차 가능성 확인

---

## 3. 앱 콘셉트

### 한 줄 설명

**목적지를 입력하면 주변 주차장의 현재·예상 주차 가능성을 점수로 보여주는 앱.**

### 서비스 이름 후보

- 파킹찬스
- 지금주차
- 주차될까
- 파킹예보
- 자리있나

---

## 4. MVP 핵심 기능

### 4.1 목적지 기반 주차장 검색

- 주소/장소명 입력
- 목적지 반경 300m, 500m, 1km 주차장 표시
- 공영/민영 구분
- 도보 거리 기준 정렬

### 4.2 주차 가능성 점수

각 주차장에 0~100점의 가능성 점수를 부여한다.

예시:

- 85점: 여유 가능성 높음
- 60점: 보통, 대안 필요
- 35점: 혼잡 가능성 높음
- 10점: 사실상 어려움

### 4.3 현재 상태 표시

가능한 경우:

- 전체 면수
- 현재 주차 가능 면수
- 만차 여부
- 최근 갱신 시각
- 요금
- 운영시간

### 4.4 도착 시점 예측

사용자가 출발 시간을 입력하거나 현재 위치를 기준으로 예상 도착 시간을 계산한다.

- 지금 출발
- 15분 뒤 도착
- 30분 뒤 도착
- 특정 시간 도착

MVP에서는 정교한 ML보다 **규칙 기반 예측**으로 시작한다.

---

## 5. 예측 로직 v0

초기 버전은 머신러닝보다 설명 가능한 점수식으로 간다.

### 5.1 입력 변수

- 현재 잔여면수
- 전체 주차면수
- 최근 데이터 갱신 시각
- 요일
- 시간대
- 공휴일 여부
- 목적지와 거리
- 주차장 유형: 공영/민영/환승/노상/노외
- 주변 POI 유형: 병원, 역, 상가, 관광지, 공연장 등

### 5.2 점수 예시

```text
availability_ratio = available_spaces / total_spaces
freshness_penalty = 데이터가 오래될수록 감점
time_risk = 시간대별 혼잡 위험
poi_risk = 주변 목적지 유형별 혼잡 위험
walk_penalty = 목적지에서 멀수록 감점

parking_score =
  availability_ratio * 60
  + freshness_score * 15
  + time_score * 15
  + distance_score * 10
  - poi_risk
```

### 5.3 표시 방식

- “가능성 높음 / 보통 / 낮음” 라벨
- 점수와 함께 이유 표시
  - 예: “현재 42면 여유, 데이터 3분 전 갱신, 목적지까지 도보 4분”

## 6. 사용할 수 있는 공공데이터 후보

> 현재 Hermes의 공공데이터 검색 MCP는 `공공데이터포털 검색 서비스(ID:15112888)` 활용 신청이 안 되어 자동 검색이 실패했다. 아래는 기획상 필요한 데이터 카테고리이며, 실제 API 명세는 별도 확인 필요.

### 6.1 필수 데이터

- 전국/지자체 공영주차장 정보
  - 위치
  - 주소
  - 운영시간
  - 요금
  - 전체 면수
  - 전화번호
- 실시간 주차 가능 면수 API
  - 지자체별 제공 여부 차이 큼
  - 서울시/부산시/대구시 등 도시별 별도 API 가능성

### 6.2 보조 데이터

- 공휴일 정보
- 날씨/강수 정보
- 행사/축제 정보
- 대중교통역 위치
- 병원/문화시설/관광지 POI
- 행정동/상권 경계

## 7. MVP 지역 전략

전국 서비스로 바로 시작하지 않는다. 실시간 주차 데이터 품질이 좋은 지역 하나를 고른다.

### 후보

1. 서울
   - 공영주차장 수요 큼
   - 실시간/공공 API 가능성 높음
   - 경쟁도 높음
2. 부산
   - 관광/도심 주차 수요 큼
   - 지역 특화 서비스로 차별화 가능
3. 대전/세종
   - 규모가 작아 MVP 검증이 쉬움
4. 제주
   - 관광객 주차 니즈 강함
   - 렌터카 사용자가 많음

### 추천 시작점

**서울 또는 제주**

- 서울: 데이터와 수요 검증에 유리
- 제주: 관광객 사용 시나리오가 선명함

## 9. 기술 구조 초안

### 9.1 프론트엔드

- React / Vite
- 지도: Kakao Maps 또는 Naver Maps
- 모바일 우선 PWA

### 9.2 백엔드

- FastAPI
- 공공데이터 API 프록시
- 주차장 데이터 정규화
- 가능성 점수 계산 API

### 9.3 데이터 저장

- MVP: SQLite 또는 Postgres
- 주차장 메타데이터 캐싱
- 실시간 상태는 짧은 TTL 캐시

### 9.4 주요 API

```http
GET /api/search?query=강남역
GET /api/parking-lots?lat=...&lng=...&radius=500
GET /api/parking-lots/{id}
GET /api/parking-lots/{id}/prediction?arrival_time=...
```

## 11. 차별화 포인트

기존 지도 앱과 차별화하려면 “정보 표시”보다 “의사결정 지원”에 집중해야 한다.

- 위치 검색이 아니라 **주차 성공 가능성 예측**
- 현재 상태뿐 아니라 **도착 시점 기준 판단**
- 만차 시 **대체 주차장 추천**
- 왜 이 점수가 나왔는지 **설명 가능한 이유 제공**
- 사용자 피드백으로 예측 개선

---

## 12. 리스크

### 12.1 데이터 품질

- 실시간 주차 가능 면수 API가 없는 지역이 많을 수 있음
- 갱신 주기가 길면 예측 신뢰도가 낮아짐
- 민영주차장 데이터 확보가 어려울 수 있음

### 12.2 지도 비용

- 상용 지도 API 호출 비용 발생 가능
- 초반에는 호출량 제한과 캐싱 필요

### 12.3 예측 책임

- “가능성”이지 “보장”이 아님을 UI에 명확히 표시해야 함
- 사용자 피드백으로 신뢰도 보정 필요
